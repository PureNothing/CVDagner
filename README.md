Запуск микросервиса VideoProduce:

    1. Запуск микросервиса backend_VideoProduce (Бекенд):
        1. cd backend_VideoProduce
        2. uv run -m app.main
        3. Эндпоинт слушает на пору 8000.
    Запустит сервер фаст апи который примет любые видео загрузить их в один бакет разделит по папкам (с какой камеры), загрузит информацию о видео с какой камеры снято, какой путь minio имеет и когда создано в БД (PostgreSQL). Нарежет видео на фреймы сложит все фреймы в отдельный бакет, со структурой папок где сначлаа каждая папка - каждая камреа, и в каждой папке, каждая папка это фейрмы конкретного видео. Отправит всю инфрмацию о нерезанных фреймах в топик кафки в микросерис детекции. И будет слушать другой топик о том что фреймы обработаны, получив такое сообщение обновит в БД на статус (processe) - обработано.
    Создаст 2 бакета в minio videos и frames. В videos будут загружены полные присланные видео. В frames будут загружены фреймы, сткрктура папок и имен файлов реализовано таким образом что у каждой камеры будет своя папка для видео, а у каждого фрейма будет своя папка с какой он камеры и папка к какому видео фрейм имеет отношение.
    Созданы 2 таблицы в отдельной схеме со связью один ко многим. Videos имеет поля во сколько загружено видео к какой камере имеет отношение, какой путь имеет в minio и к каким фремам имеет отношение (одно видео ко многим фреймам). Frames имеет поля во сколько добавлен после нарезки фрейм, к какой камере имеет отншение, какой путь имеет в минио, к какому видео имеет отношение (много фреймов к одному видео) и статус обработки (по умолчаению not_processed после слушате очеред из сервиса детекции и меняет на процессед (обработан))

        2. Слушает на порту 8501

Запуск микросервиса DetectGraphApi:
    1. cd DetectGraphApi
    2. uv run -m app.main
    3. Эндпоинт слушает на порту 8002.
Запустит сервер который будет ждать сообщения из очереди о нарезанных фреймах, возьмет каждый фрейм и детектирует его (отправив на сервер детекции с моделью YOLOE заточенной файн тюном под 8 классов). Этот сервер вернет результаты детекции на основании которых на каждом фрейме будут нарисованы боксы и подписаны классы, все это будет загружено в отдельный бакет Minio (detected). Если на фреймах ничего нет они не будут грузится в бакет. После распознавания отправит инормацию о распознанных объекта и их колличестве на каждом фото в сервис алертов AlertsService.
Имеет внутри 2 таблицы в отедьной схемы базы данных со связью (one-to-many), таблица Detected frames хранит информацию о каждой обрабтанном фрейме замечено нам нем было что то или нет, во сколько обработан, к какой камере отностися, какой путь к файлу в Minio и связь с таблицей детекий (один фрейм к многим объектам). Таблица Detected Objects хранит каждый детектированный объект имеет связь с таблицей фреймов (многие объекты к одному фрейму), камеру на которой замечен объект, его класс, уверенность в детекции, бокс детекции и во сколько был замечен объект.
Далее в этом сервисе реализована GraphQL схема с запросами к БД и этим таблицам и гибкими ответами. Вся эта логика GraphQL запросов в итоге реализована кнопках бота. Отчет по каждой камере за 24 часа, общий отчет по всем камерам за всё время и самая опасная камера, реализовано это за счет подчета объектов каждого класса, в интервале 24 часа , либо за всё время наибольшее колличество обнаруженных объектов (сгруппировано по каждой камере).

Запуск микросвервиса AlertsService:
    1. cd AlertsSevice
    2. uv run -m app.main
    3. Эндпоинты слушают на порту 8002.
Запустит сервер который ждет сообщения о всех детекция, имеет внутри Json файл, по каждой камере с описанием местности возле камеры, коордиантами камеры и порогами для алертов каждой камеры. При условии что порог превышен, отправит все превышение пороги в AIService для принятия решения по алерту, о этом дальше.
Так же есть дополнительныый функционал через бота и AI сервис.
    1. Изменения порога любой камеры и любого класса по свободному разговоному сообщения пользователя (парсится SO LLM)
    2. Добавление новой камеры с любым номером, любыми координатами и описанием местности по свободному сообщению пользователя (Парсится SO LLM)

Запуск микросервиса AIService:
    1. cd AISerivice
    2. uv run -m app.main
    3. Эндпоинты слушают на порту 8003.
Запустит серверв с AI-агентами которыеы слушают очередь алертов, выполняют регулярные задачи, парсят сообщения.
    Агенты:
        1. При получение алерта из AlertsService будет должен принять решение с выбором инструментов (ToolCalling) на основании местности вокруг камеры что ему использовать (Отправить разведывательного дрона или авиаудар). Сообщит о выборе пользователю в боте. Прислав уведомление что был использован инструмент по камере с указнными номером, координатами и местностью.
        2. Получит сообщение из бота о добавление новой камеры в свободной форме, сам распарсит его, провалидирует (Structured Output) и отправит в сервис AlertsService для изменения Json файла добавив камеру.
        3. Получит сообщения из бота об изменении порога в свободно форме, сам распарсит его, провалидирует (SO) и отправит в сервис Alerts Service для изменения порога по конкретному классу для конкретной камеры.
        4. Получит Shudule Task из бота (каждые 10 минут) с полным отчетом по камерам и должен будет проанализировать его, составив свой отчет и предоставив пользователю (Вернет в бота свое мнение по каждой камере и советы для безопасности).

Запуск микосервиса Telegram_bot:
    1. cd TgGraphBot
    1. uv run -m bot.main
    2. Перейти в бота и пользоаться
Запустит телеграм бота объеденив весь предудщий пайплайн и замкнув кольцо, помимо этого запусти Celery worker и beat, для формирования отложенных задач и выполнения их, все отложенные задачи будет грузить в redis.
Бот имеет такой функционал:
    Кнопка "Помощь" - для объяснения и введение в весь пайплайн.
    Кнопка "Главное меню" которая выводит список кнопок (кнопка камер от 1 до 5 и кнопка полный отчет, все кнопки интерактивно украшены и нативно понятны) под капотом каждой кнопки реализована логика обращений graphql к сервису детекций через эндпоинт DetectGraphApi к его базам данным для получения актуальной информации, помимо этого под каждой кнопкой реализована логика обращения к сервису AlertsService через его эндпоинты для получения актуальных коордиант и местности для каждой камеры в отчете.
    Отчет по каждой отедьной камере предоставить все замечанные классы на камере за последние 24 часа и в каком колличестве они были замечаны, коордианаты каммеры и её местность.
    Полный отчет представит отчет по каждой камеры за всё время а не 24 часа, все детектированные классы, их колличество, коордианаты камеры и её местность.
    Помимо этого у бота есть дополнительная кнопка меню настроек которая позволяет пользователю который имеет скретеный ключ (проверка по кторому призводится) получит доступ к глубоким настройкам, все команды и сообещния настройек доступны только в состояние активной фазы настроек которую получаем после ввода ключа, после использования какой то опции из глубоких настроек доступ сбрастываеся и пользователю нужно ввести секретный ключ снова. Имеются 2 опции. Для каждой из них приручено понятное описание функционала и возможностей.
        1. Добавление новой камеры. Пользователь в свобондном языке не структурированно может сказать какую камеру добавить с каким номерам какими координатами и какая радом местность, всё это уйдет в сервис AIService из этого сделается стркутрунй вывод а потом это уйдет и в AlertsService который и изменит Json по соотвествующей структуру добавив камеру со всеми необходимыми данными.
        2. Измения порога имеющейся камеры. Пользователь в свободном языке не сркутрурировано сообщает для какой камеры и для какого класса на этой камере хочет изменить порог. Это сообщеия отправится в AIService там сделается стркутурный вывод извлекутся факты и отправится в сервис с правилами AlertsService, там уже изменится порог и соотвественно алерты будут приходить уже по другому порогу.
    Помимо этого каждый 10 минут с помощью celery бот формируте отложенную задачу полного отчета через LLM, он присылает в AIService полный отчет который должен проианализировать бот и представить ещё более полный отчет с мнением бота по каждой камере, общем состоянием системы и безопасонсти и советами на что обратить внимание, пользователь получит этот отчет в чат.
