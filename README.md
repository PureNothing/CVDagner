# CVDanger — Система видеонаблюдения с детекцией военных объектов

CVDanger — это микросервисная система реального времени для автоматической детекции военной техники и пехоты на видео с камер наблюдения. Система принимает видео, нарезает его на фреймы, прогоняет через YOLO модель, анализирует угрозы, принимает решения с помощью AI и уведомляет оператора через Telegram.

---

## Общий пайплайн

```
Оператор загружает видео
        ↓
backend_VideoProduce    — принимает видео, нарезает на фреймы, складывает в MinIO
        ↓ Kafka: ready_to_detect
DetectGraphApi          — детектирует каждый фрейм через YOLO, рисует боксы, сохраняет в БД
        ↓ Kafka: alerts_topic
AlertsService           — проверяет пороги, если превышены — формирует алерт
        ↓ Kafka: real_alerts
AIService               — AI агент принимает решение (дрон / авиаудар), уведомляет бота
        ↓ Kafka: agent_actions
telegram_bot            — отправляет алерты и решения агента оператору
```

Параллельно:
- `telegram_bot` каждые 10 минут автоматически запрашивает полный отчёт и отправляет AI-анализ оператору
- Оператор через бота может менять пороги алертов и добавлять новые камеры в свободной форме — это уходит в AIService → AlertsService
- Оператор через бота запрашивает отчёты по каждой камере — данные берутся из DetectGraphApi через GraphQL

# BentoService — YOLO детектор

BentoService — это изолированный ML сервис, единственная задача которого принять изображение и вернуть результаты детекции. Остальные микросервисы не знают ничего о модели — они просто шлют сюда base64 картинку и получают боксы с классами.

Внутри работает YOLOE модель, дообученная на ~3000 фотографий военной техники и пехоты. Модель сконвертирована в ONNX формат для стабильного и быстрого инференса.

## Детектируемые классы

Пехота, Танк, БМП (Боевая Машина Пехоты), БТР (Бронетранспортер), Бронемашина, Артиллерия, РСЗО (Ракетная Система Залпового Огня), БПЛА (Беспилотный Летательный Аппарат).

Порог уверенности по умолчанию — `0.4`. Всё ниже этого значения игнорируется.

## Как работает

Сервис поднимается на порту `7777` и принимает `POST /detect` с base64 изображением. Возвращает список боксов, классов и уверенности. Если ничего не найдено — возвращает пустые списки. DetectGraphApi вызывает этот эндпоинт для каждого фрейма.

## Подготовка модели

Прежде чем собирать контейнер, модель нужно зарегистрировать в BentoML store. Для этого положи `.onnx` файл в `BentoService/FineTune/KaggleModel/`, укажи его имя в `bentomodel.py` и запусти:

```bash
uv run BentoService/bentomodel.py
```

## Сборка контейнера

```bash
# Шаг 1 — сборка bento-архива
uv run bentoml build -f bentofile.detect.yaml

# Шаг 2 — контейнеризация
uv run bentoml containerize war-detector:latest --image-tag war-detector:latest

# Шаг 3 — обязательный патч OpenCV
docker build -f dockerfile.fix -t war-detector:fixed .
```

Шаг 3 обязателен: в `opencv-python==4.13.0.90` есть баг — при импорте `cv2` в контейнере падает `ImportError: libxcb.so.1`. Причина в том что bundled библиотеки этой версии имеют неразрешённые зависимости на системный `libxcb.so.1`, которого нет в минимальных Docker образах. `dockerfile.fix` откатывает OpenCV на стабильную версию `4.12.0.88`. ([GitHub issue](https://github.com/opencv/opencv-python/issues/1183))

> ⚠️ Перед сборкой убедись что в `modelservice.py` аннотация параметра `bytes`, а не `Image` — иначе контейнер не соберётся.

## Режимы работы модели

Сервис поддерживает два режима — переключение через `modelservice.py`:

**Файнтюн модель (текущий, рекомендуемый)** — активна строка `labels.append(FINETUNED_MAP[cls_id])`. Использует дообученную модель с фиксированным маппингом классов по индексу.

**Базовая YOLOE модель (zero-shot)** — нужно раскомментировать `self.model.set_classes(self.classes)` и соответствующую строку маппинга. Работает без файнтюна, но хуже распознаёт военную технику.

## Файнтюн своей модели

Если нужно обучить модель на своём датасете:

**Локально:**
1. Загрузи изображения в `yolo_dataset/images/train/` и CSV разметку в `yolo_dataset/csv/`
2. Замени классы в `convert_csv_to_yolo.py` и `dataset.yaml` на свои
3. Конвертируй разметку: `uv run BentoService/FineTune/convert_csv_to_yolo.py`
4. Запусти обучение: `uv run BentoService/FineTune/fine_tune.py`
5. Лучшие веса появятся в `runs/train/yolo_finetuned/weights/best.pt`

**Kaggle / Google Colab** (рекомендуется при слабом железе): перенеси код на платформу, загрузи датасет и веса, обучи на GPU, скачай `best.pt` и положи в `KaggleModel/`.

После обучения сконвертируй модель в ONNX для быстрого инференса:

```bash
uv run BentoService/FineTune/konvert_to_onnx/convert_to_onnx.py
```

# backend_VideoProduce — Приём и нарезка видео

Это точка входа видео в систему. Сервис принимает видеофайлы от оператора, сохраняет их в MinIO, нарезает на фреймы и запускает пайплайн детекции через Kafka.

## Как работает

Оператор отправляет POST запрос с видеофайлами и номером камеры. Для каждого видео сервис генерирует уникальный UUID, сохраняет оригинал в MinIO бакет `videos` и создаёт запись в БД. Затем видео нарезается на фреймы — берётся один кадр в секунду. Каждый фрейм загружается в бакет `frames`, записывается в БД со статусом `not_processed` и путь к нему публикуется в Kafka топик `ready_to_detect`.

Параллельно сервис слушает топик `frame_detected` — когда DetectGraphApi заканчивает обработку фрейма, он сообщает об этом сюда, и статус фрейма в БД меняется на `processed`.

## Структура хранения в MinIO

Все файлы организованы так, чтобы по пути можно было восстановить связь между видео, камерой и фреймами:

```
videos/  camera_id=2/550e8400-uuid.mp4
frames/  camera_id=2/550e8400-uuid/frame00000.jpg
         camera_id=2/550e8400-uuid/frame00025.jpg
```

Видео и все его фреймы разделяют один UUID — это связывает их без дополнительных запросов.

## База данных

Схема `VideoProducerService` содержит две таблицы со связью один-ко-многим.

**Videos** — одна запись на каждое загруженное видео: время загрузки, номер камеры, путь в MinIO.

**Frames** — одна запись на каждый нарезанный фрейм: время нарезки, номер камеры, путь в MinIO, ссылка на видео, статус обработки (`not_processed` / `processed`).

> ⚠️ При каждом старте сервиса таблицы пересоздаются — все данные удаляются.

## Известные ограничения

Если публикация фрейма в Kafka упала с ошибкой — фрейм остаётся в MinIO и БД, но DetectGraphApi его не увидит и детекция не произойдёт. Ошибка логируется но не пробрасывается.

## API

`POST /video_upload/{camera_id}` — принимает один или несколько видеофайлов через multipart/form-data.

# DetectGraphApi — Детекция и GraphQL API

Центральный сервис обработки. Слушает очередь готовых к детекции фреймов, прогоняет каждый через BentoService, сохраняет результаты и предоставляет GraphQL API для получения отчётов.

## Как работает

Сервис слушает Kafka топик `ready_to_detect` с параллельностью 2 воркера. Для каждого фрейма:

1. Скачивает оригинал из MinIO бакета `frames`
2. Кодирует в base64 и отправляет в BentoService на детекцию
3. Если YOLO нашла объекты — рисует боксы поверх изображения через PIL и загружает результат в бакет `detected`. Путь к файлу с боксами получается добавлением `_detected` к имени оригинала
4. Сохраняет в БД запись о фрейме и каждом найденном объекте отдельно
5. Если объекты найдены — публикует результаты в `alerts_topic` для AlertsService
6. В любом случае публикует путь фрейма в `frame_detected` чтобы backend_VideoProduce обновил статус

Если на фрейме ничего нет — он всё равно записывается в БД, просто без пути к файлу с боксами и без отправки в `alerts_topic`.

> ⚠️ `drawer.py` использует шрифт `arial.ttf`. На Linux он может отсутствовать — положи файл в корень `DetectGraphApi/` перед запуском.

## База данных

Схема `DetectionService` содержит две таблицы со связью один-ко-многим.

**Detected_frames** — одна запись на каждый обработанный фрейм: номер камеры, путь к оригиналу, путь к версии с боксами (или `None` если ничего не найдено), время обработки.

**Detected_objects** — одна запись на каждый найденный объект: ссылка на фрейм, номер камеры, класс объекта, уверенность детекции, координаты бокса (`x1, y1, x2, y2`), время детекции.

> ⚠️ При каждом старте сервиса таблицы пересоздаются — все данные удаляются.

## GraphQL API

Сервис поднимает GraphQL эндпоинт на `http://localhost:8001/graphql` с интерактивным IDE прямо в браузере. Именно через него бот запрашивает все отчёты.

**Отчёт по одной камере** (`getCameraReport`) — возвращает все задетектированные классы за последние 24 часа, их количество и время последнего обнаружения. Если за 24 часа ничего не было — возвращает `isDangerDetected: false`.

**Общий отчёт** (`getGeneralReport`) — возвращает статистику по всем камерам за всё время и выделяет самую опасную камеру по общему количеству детекций.

# AlertsService — Пороги и управление правилами

Этот сервис — фильтр между детекцией и реакцией. Он не просто пересылает все детекции дальше, а проверяет: достаточно ли угроз на этом фрейме чтобы поднимать тревогу. Конфигурация всех камер хранится в `rules.json`.

## Как работает

Сервис слушает `alerts_topic` с результатами каждой детекции. Получив сообщение, он читает из `rules.json` пороги для этой камеры и считает сколько объектов каждого класса найдено на фрейме. Если количество объектов какого-то класса достигает или превышает порог — публикуется алерт в `real_alerts`. Каждый превышенный класс — отдельное сообщение, чтобы AIService мог принимать решение по каждому независимо.

В алерт добавляются координаты камеры и описание местности вокруг неё — именно на основании этого AIService будет принимать решение что делать.

## rules.json

Единственный источник конфигурации сервиса. Хранит по каждой камере: координаты (lat, lon), описание местности вокруг камеры и пороги срабатывания для каждого из 8 классов. По умолчанию настроены камеры 1–5 с порогом 2 для всех классов.

Файл изменяется динамически — через API сервиса когда оператор меняет пороги или добавляет камеры через бота.

## API

Все эндпоинты используются AIService и ботом для управления конфигурацией:

- `POST /change_rules` — изменить порог для конкретного класса на конкретной камере. При изменении через бота сообщение оператора сначала парсится LLM в AIService, и уже структурированные данные приходят сюда
- `POST /add_camera` — добавить новую камеру с координатами и описанием местности. Новая камера получает порог 2 для всех классов по умолчанию
- `POST /change_coordinates` — обновить координаты камеры
- `GET /get_coordinates/{camera_id}` — вернуть координаты (используется AIService при принятии решения об ударе)
- `GET /get_restrictions/{camera_id}` — вернуть описание местности (используется аналогично)

При изменении порога входящий `label` проверяется по жёстко заданному списку допустимых классов — это защита от некорректных данных от LLM.

# AIService — AI агенты

Сервис с четырьмя AI агентами на базе Groq API. Каждый агент решает свою задачу: принимать решения при угрозах, парсить свободный текст от оператора, анализировать общую обстановку.

## Агент принятия решений (Tool Calling)

Когда AlertsService фиксирует превышение порога — алерт уходит в `real_alerts` и этот агент его получает. Агент знает что именно замечено (класс и количество) и какая местность вокруг камеры. На основании этого он выбирает один из двух инструментов: отправить разведывательного дрона или вызвать авиаудар.

Логика выбора заложена в промпт: наша территория или много мирных — только дрон; территория врага без мирных — можно авиаудар; если техники больше 2 или пехоты больше 5 на территории врага где есть мирные — тоже авиаудар. Итоговое решение принимает LLM взвешивая все факторы.

После выбора инструмент запрашивает точные координаты камеры у AlertsService и публикует результат в `agent_actions` — бот получит уведомление о том что было сделано и где.

## Агент парсинга (Structured Output)

Когда оператор пишет боту в свободной форме — например _"на второй камере для танков поставь порог пять"_ — это сообщение приходит сюда. Агент извлекает три факта: номер камеры, класс объекта и новый порог, возвращает структурированный объект и сервис отправляет его в AlertsService для изменения `rules.json`. Аналогично работает парсинг при добавлении новой камеры — извлекается номер, координаты и описание местности.

При ошибке парсинга агент возвращает строку с просьбой переформулировать — бот покажет её оператору.

## Агент аналитика (Report)

Каждые 10 минут бот собирает полный отчёт по всем камерам и отправляет его сюда. Агент получает сырые данные детекций и составляет структурированный военный доклад: общая обстановка, анализ по каждой камере, тенденции и паттерны угроз, конкретные рекомендации. Доклад уходит обратно боту и отправляется оператору.

# telegram_bot — Интерфейс оператора

Бот замыкает весь пайплайн — он одновременно получает уведомления из Kafka и предоставляет оператору интерфейс для запросов и управления.

## Как устроен

При запуске бот поднимает три параллельных процесса в отдельных потоках: Kafka consumer для получения алертов и действий агента, Celery worker для выполнения отложенных задач, Celery beat для планировщика (каждые 10 минут).

Два оператора (`OWNER_ID` и `BUDDY_ID`) получают уведомления об алертах и действиях агента. Автоматический 10-минутный отчёт уходит только `OWNER_ID`.

> ⚠️ **Баг:** сейчас оба `send_message` в `kafkaconsume.py` и `kafkaconsumeactions.py` отправляют на `OWNER_ID` — `BUDDY_ID` ничего не получает. Нужно поменять второй вызов на `BUDDY_ID`.

## Функциональность

**Отчёты по камерам** — кнопка «Главное меню» показывает 5 кнопок камер и кнопку полного отчёта. При нажатии на камеру бот параллельно запрашивает GraphQL отчёт из DetectGraphApi и координаты/местность из AlertsService, форматирует и отправляет. Отчёт по отдельной камере — за 24 часа, полный отчёт — за всё время.

**Настройки** (доступны только после ввода секретного ключа):
- Изменить порог алерта — оператор пишет в свободной форме, AIService парсит и меняет в AlertsService
- Добавить камеру — аналогично, свободный текст → AIService → AlertsService

После каждого действия в настройках доступ сбрасывается и нужно вводить ключ снова.

**Автоматический отчёт** — каждые 10 минут Celery запускает задачу: собирает GraphQL отчёт по всем камерам + координаты и местность всех 5 камер параллельно, форматирует, отправляет в AIService для анализа, готовый доклад уходит оператору в чат. При сбое — 5 попыток повтора с нарастающей задержкой.

## Известные ограничения

Кнопки камер (1–5) и полный отчёт захардкожены. Если добавить новую камеру через настройки — она появится в AlertsService и будет детектировать, но кнопки в боте и 10-минутный отчёт её не увидят.

| Топик | Кто публикует | Кто слушает | Что передаётся |

## PostgreSQL — схемы и таблицы

В одной БД два микросервиса используют отдельные схемы:

**Схема `VideoProducerService`** (backend_VideoProduce):
- `Videos` — загруженные видео: камера, путь в MinIO, время загрузки
- `Frames` — нарезанные фреймы: камера, путь в MinIO, ссылка на видео, статус (`not_processed` / `processed`)

**Схема `DetectionService`** (DetectGraphApi):
- `Detected_frames` — обработанные фреймы: камера, путь к оригиналу, путь к версии с боксами (или None), время обработки
- `Detected_objects` — найденные объекты: ссылка на фрейм, камера, класс, уверенность, координаты бокса, время

> ⚠️ Обе схемы пересоздаются при каждом старте соответствующего сервиса — данные удаляются.
